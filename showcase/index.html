1) Why We Developed a Big Data SIEM?

In modern enterprise networks processing millions of events per second (EPS), traditional SIEMs collapse under astronomical licensing fees and severe hardware constraints. This open-source Big Data SIEM architecture is designed to break these barriers and solve the logging bottlenecks:

  • Cost-Effective Scalability: Leverages distributed open-source components to maximize hardware utilization and bypass licensing limits.
  • Unlimited Distributed Storage: Employs Apache Hadoop HDFS to securely persist petabytes of chronological log archives.
  • Hot/Cold Data Tiering: Performs sub-second analysis in RAM for active logs, archiving historical trails in the Hive Metastore.
  • Hybrid Detection Model: Runs both deterministic SIEM rules and probabilistic Machine Learning anomaly models concurrently.

3) Real-World Ingestion: SecRepo Cybersecurity Datasets

For high-fidelity architectural validation, we bypass synthetic laboratory networks. Instead, our platform ingests real-world enterprise traffic logs from the SecRepo.com repository containing organic scans and penetration indicators:

  • Active Zeek Network Telemetry: Ingests continuous conn.log (524 MB connection states) and http.log (54 MB transactions) alongside `dns`, `ssl`, and `weird` anomaly signals.
  • Integrated Snort Alarms: Maps and correlates Snort threat signature files (maccdc2012_fast_alert and maccdc2012_full_alert) in parallel.
  • Why Exclude PCAP Files? Raw PCAP processing imposes high packet decoding overhead. Streaming structured Zeek and Snort text lines directly preserves sub-second analytical latencies in the distributed ingestion pipe.

4) Ingestion Layer: Multi-Stream Kafka Producer Container

The gateway of our analytical platform is a specialized Kafka Producer container engineered for stability and low latency:

  • Direct Volume Mount: Scans the local /data path read-only, fetching massive log tables (conn, http, dns, ssl) without disk footprint copies.
  • Throttled Traffic Simulation: Exposes configurable emission rates (e.g., `2,000` rows/second) to stresses-test the Spark consumer under steady volumetric loads.
  • Intelligent Topic Routing: Automatically maps specific network logs to target broker pipelines (real_network_logs, web-logs, dns-logs, ssl-logs), converting historical databases into active streams.

5) Ingestion Buffering: Apache Kafka (KRaft Mode)

Apache Kafka functions as the platform's high-speed nervous system, buffering volumetric events and protecting the downstream analytics core during severe DDoS attacks:

  • KRaft (ZooKeeperless Raft Mode): Eliminates external synchronization clusters, managing internal coordination directly in the JVM for rapid failover.
  • Resilient Backpressure Safeguard: Retains high-volume spikes in sequential disk offsets, preventing memory exhaustion on analytical workers.
  • Balanced Partition Routing: Parallelizes event reads across distinct partition streams, guaranteeing a uniform workload across Spark worker nodes.

6) Analytics Engine: Apache Spark Structured Streaming

Apache Spark acts as the analytical core of our architecture. Working entirely in-memory, it parses, transforms, and runs K-Means anomaly boundaries at sub-second speeds:

  • Structured Streaming: Implements declarative, fault-tolerant schema-on-read pipelines integrated directly with the Spark SQL catalyst compiler.
  • Micro-Batching Execution: Periodically queries active Kafka offsets (e.g., every 5-10 seconds), running complex threat checks across millions of rows in RAM.
  • Project Tungsten and Catalyst: Bypasses standard JVM heap limitations using off-heap allocations, optimizing CPU cache locality for rapid data processing.

7) Distributed Storage: Hadoop HDFS Data Lake

To archive massive event trails safely, the platform distributes raw and parsed logs across decentralized storage blocks using Hadoop Distributed File System (HDFS):

  • NameNode/DataNode Isolation: Records system directories and cluster allocations on the NameNode, keeping the actual raw blocks distributed on DataNodes.
  • Reliable Block Replication: Clones blocks across multiple DataNodes (2x replication) to protect threat archives against sudden physical drive crashes.
  • Scale-Out Flexibility: Horizontally extends storage space on-the-fly by joining new DataNode containers without disrupting active streaming queries.

8) Structured Warehousing: Apache Hive & Parquet

Apache Hive bridges the gap between raw block files and structured database queries, exposing the HDFS data lake to standard SQL interfaces:

  • Hive Metastore (HMS): Manages schemas and maps physical directory paths, sharing catalog configurations with both Spark and Superset.
  • Columnar Parquet Format: Structures records into columns instead of rows. Threat queries reading specific columns (e.g., `src_ip`) completely skip other fields, minimizing I/O bottlenecks.
  • Dynamic Date Partitioning: Organizes directories based on date keys (ingest_date). Historical audits prune away unrelated folders instantly, accelerating queries logarithmically.

9) Interactive Threat Hunting: Apache Zeppelin

An interactive, notebook-style analytical playground that allows SOC threat hunters to explore billions of records in seconds using Spark SQL:

  • Spark SQL Interpreter: Runs direct, complex ANSI SQL queries across historical Hive data lake tables with zero environment setup.
  • Parametric Query Binding: Exposes interactive form widgets to let analysts filter queries by custom IP scopes or port parameters on the fly.
  • Instant Visual Triage: Plots large query outputs into clean bar graphs, pie charts, and chronological trends directly inside the notebook.

10) SOC Dashboard & Analytics: Apache Superset

Our real-time Security Operations Center (SOC) dashboard. It maps incoming logs, Spark ML classifications, and threat intelligence events on unified screens connected via JDBC to the Spark Thrift server:

  • Real-Time Threat Visualization: Monitors system health metrics, incoming event rates (EPS), and active alarms in real-time.
  • 12-Column Responsive Layout: Expands and organizes charts to fit widescreen displays, presenting comprehensive forensic details (IPs, ports, anomaly scores).
  • GeoIP Heatmaps: Enriches external IP signals with geographic coordinates, plotting international attacks on an interactive world map for quick triage.

11) Performance Hardening: PostgreSQL & Redis Cache

Infrastructure accelerators backing our distributed SIEM to guarantee low-latency dashboard loads and stable metadata tracking:

  • PostgreSQL Metastore (ACID Core): Formats and tracks table schemas for the Hive Metastore and persists user credentials and widget settings for Apache Superset.
  • Redis LFU Cache: Caches repetitive analytical queries from concurrent SOC dashboards, serving recurring charts in milliseconds rather than query cold storage.
  • Reduced Data Lake Load: Protects Hive and HDFS disks from query saturation during high analyst concurrency.

12) Threat Context: Contextual Feature Engineering

To eliminate false positives, our streaming pipeline enriches network logs with crucial security context before training the unsupervised model:

  • Why Exclude Raw IPs? IP strings are categorical descriptors, not continuous values. Feeding raw IPs directly into Euclidean distance calculations is a mathematical violation that triggers the curse of dimensionality.
  • Contextual PySpark Flags: Replaces IP addresses with numeric binary indicators (0/1) extracted via high-speed regex logic in the Spark ingestion stream:
  • 1. External Target Flag (is_dst_external): Evaluates if destination IPs resolve outside RFC1918 private subnets (e.g., 10.0.0.0/8, 192.168.0.0/16).
  • 2. Server Role Flag (is_src_server): Identifies if the source host matches enterprise servers. This ensures massive, legitimate backup traffic is separated from external data exfiltration anomalies.

13) Cyber Intelligence: Machine Learning Mathematical Model

The mathematical pipeline executed inside Spark Structured Streaming to classify standard activities and isolate outlying exceptions:

  • Objective Function (WSSSE): Assigns $n$ flow coordinates to $k=8$ centroids ($\mu_i$) to minimize variances:
    $$\arg \min_S \sum_{i=1}^k \sum_{x \in S_i} \| x - \mu_i \|^2$$
  • StandardScaler Scaling: Normalizes features to $Mean=0$ and $Variance=1$ to prevent large byte fields from dominating brief connection durations.
  • Euclidean Distance Metric: Calculates the real-time distance ($d$) of new flows ($f$) to their assigned cluster centroid ($c$):
    $$d(f, c) = \sqrt{\sum_{j=1}^m (f_j - c_{cluster\_id, j})^2}$$

14) Adaptive Intelligence: Asynchronous In-Memory Retraining

Network baselines evolve over time (Concept Drift). To prevent the SIEM from becoming blind to new behaviors, the pipeline executes asynchronous, continuous model retraining in memory:

  • Bootstrap Buffering (Cold Start): Holds initial streaming logs (COLD_START_ROWS = 100) to compute baseline starting clusters.
  • Periodic Model Re-training: Queries the cumulative parquet warehouse every 50 batches (RETRAIN_EVERY_BATCHES = 50), recalculating centroid coordinates on modern data.
  • Adaptive Shielding: centroids adjust dynamically to changing traffic profiles, keeping classification boundaries highly accurate without admin intervention.

15) Cyber Intelligence: Outlier Clustering (2D Projection)

A 2D mathematical projection demonstrating how our trained K-Means model isolates malicious outliers from baseline traffic patterns:

  • Green Clusters (Baseline Traffic): Legitimate connection streams tightly bunched around their nearest centroid.
  • Red Outliers (Anomalous Spikes): Highly anomalous sessions falling past the standard deviation limits, representing rapid port scans or heavy exfiltration channels.
  • Parallel Inferences: Centroid distance calculations execute natively across Spark executor cores, maintaining sub-second ingestion rates.

16) Model Validation: WSSSE Convergence & Metrics

Tracking training metrics and within-cluster variance to guarantee model stability under massive event volumes:

  • Mathematical Anomaly Score: The direct distance ($d$) of a vector from its nearest centroid matches its continuous Anomaly Score.
  • Outlier Threshold Limit: Events with scores exceeding a standard deviation boundary ($d \ge 3.0$) are isolated as outliers, triggering HDFS alert logs.
  • WSSSE Logging: Stores Within-Cluster Sum of Squared Errors (WSSSE) directly inside the siem.model_metrics catalog to monitor model convergence.

17) Hybrid In-Memory Model: Deterministic SIEM Rules

Predefined threat rules process active event streams in parallel to trap signature-based scans and volumetric indicators in seconds:

  • SIEM-001 (Port Scans): Identifies single client hosts querying more than 50 distinct destination port channels within a 10-second window.
  • SIEM-002 (Path Traversal): Leverages regex to spot directory traversal queries (../, etc/passwd, win.ini) inside HTTP strings.
  • SIEM-003 (Volumetric Floods): Triggers alert triggers if a single source IP mounts more than 100 connections in a single micro-batch.
